# AI Agent 系统学习讲义

---

## 第一部分：核心概念体系

### 1.1 基础概念

| 概念 | 定义 | 关键点 |
|:---|:---|:---|
| **Prompt** | 发送给大语言模型的指令或问题 | AI 交互的基本单位 |
| **User Prompt** | 用户输入的自然语言指令 | 即用户的原始问题 |
| **System Prompt** | 预设的结构化指令，定义 AI 的角色、规则和输出格式 | 常用 JSON 格式，优先级高于 User Prompt |
| **Token** | 大语言模型处理文本的最小单元 | 计费单位；上下文长度限制单位 |
| **Context** | 一次对话中模型能"看到"的所有历史信息 | 决定模型的"记忆"范围 |
| **Context Length** | 模型的最大上下文窗口大小 | 以 Token 数表示，如 128K、200K |
| **KV Cache** | 缓存已计算的 Token 键值对，加速生成 | 显存消耗与 Context Length 正相关 |

### 1.2 进阶概念

| 概念 | 定义 | 应用场景 |
|:---|:---|:---|
| **RAG** | 检索增强生成：先从知识库检索相关信息，再让模型基于这些信息生成答案 | 解决模型知识陈旧、幻觉问题；构建企业知识问答系统 |
| **Function Calling** | 让模型调用外部函数/API 的能力 | 查天气、发邮件、操作文件、调用数据库 |
| **Skill** | 将 System Prompt、Function Calling 和业务逻辑封装成可复用的"技能包" | 规范化 AI 行为；知识复用；团队协作 |
| **Agent** | 能够感知环境、自主决策、调用工具完成复杂任务的智能程序 | 自动化工作流；多步推理；任务规划与执行 |

### 1.3 模型架构参数

| 参数 | 含义 |
|:---|:---|
| **XXB** | 模型参数量，如 7B、13B、35B。越大通常越智能，资源消耗也越大 |
| **MoE 架构** | 混合专家模型。如 Qwen2.5-35B-A3B：总参数 35B，每次激活约 3B，平衡性能与效率 |
| **量化** | 压缩模型权重精度以减小体积。Q4（4bit）最小最快，智能略有下降；Q6 更平衡；Q8/F16 精度最高 |

## 第二部分：提示词工程

### 2.1 基础技巧

#### 角色定义

- **指定 AI 角色身份**：`"你是一位资深法律顾问，擅长合同法"`
- **描述用户自己的身份**：`"我是一个初创公司创始人，不懂法律术语"`

#### 输出控制

- **指定回答模板**：`"请按以下格式回答：问题重述 → 分析过程 → 结论 → 建议"`
- **反向约束**：`"请用段落叙述，不要使用表格或列表"`

#### 对抗式框架

让两个 AI 角色辩论，从对立观点中提炼更全面的结论：

> 你扮演正方，认为远程办公利大于弊；我扮演反方，认为弊大于利。
> 请开始辩论，每轮各说一个观点，3 轮后总结出客观结论。

#### 上下文管理

- **手动构造上下文**：提问前先提供相关背景信息
- **压缩上下文**：当对话过长时，用摘要压缩历史信息

> "请用 200 字总结我们刚才讨论的所有要点，保留关键决策和结论。"

### 2.2 进阶技巧

#### AI 主动询问细节

- **一次性多问题**：

> "为帮你制定健身计划，请回答：
> 1. 你的身高体重？
> 2. 每周可用天数？
> 3. 主要目标（减脂/增肌/保持）？
> 4. 有无伤病史？"

- **选择题引导**：

> "你更喜欢哪种运动方式？
> A. 有氧跑步
> B. 力量训练
> C. 团体课程
> D. 居家跟练"

#### 提示词生成提示词

给 AI 一个示例（文本、图片描述、网页），让它复刻风格、结构或逻辑：

> "这是我喜欢的几段文案，请分析其风格特点，然后按相同风格帮我写一段
> 关于'清晨咖啡'的文案：[附上示例]"

### 2.3 输出格式控制

| 格式 | 适用场景 | 示例指令 |
|:---|:---|:---|
| 段落叙述 | 需要连贯论证、故事叙述 | "请用自然段落形式叙述" |
| 表格 | 数据对比、信息整理 | "用表格对比三种方案的优缺点" |
| 列表 | 要点罗列、步骤说明 | "用无序列表列出操作步骤" |
| JSON | 供程序解析的结构化输出 | "以 JSON 格式返回，字段包括 name, age, city" |

## 第三部分：开发环境与基础调用

### 3.1 大语言模型获取途径

#### 本地部署（LM Studio）

- **优势**：数据隐私、无网络依赖、长期成本低
- **条件**：NVIDIA 显卡优先，显存大小决定可运行模型
- **选择原则**：模型文件大小 < 可用显存

#### 在线 API

- **优势**：无需昂贵硬件、模型强大、即开即用
- **推荐平台**：OpenRouter.ai（聚合多模型）、各厂商官网
- **成本控制**：利用免费额度、使用小模型做实验

### 3.2 开发环境配置

| 组件 | 说明 | 版本要求 |
|:---|:---|:---|
| Python | 主开发语言 | 3.12+ |
| AI Coding 平台 | Trae / VS Code / Cursor | 最新版 |
| Git | 版本控制 | 最新版 |
| 虚拟环境 | venv / conda | 每个项目独立 |

### 3.3 环境变量管理

创建 `.env.example` 模板文件（提交到 Git），用户复制为 `.env`（不提交）并填写真实信息：

```bash
# .env.example 模板
BASE_URL=http://localhost:1234/v1
API_KEY=your-api-key-here
MODEL=local-model-name
```

### 3.4 基础 API 调用流程

1. 读取 `.env` 配置
2. 构造请求 Headers（`Authorization: Bearer {API_KEY}`）
3. 构造请求 Body（`model`, `messages`, `stream` 等参数）
4. 发送 HTTP 请求
5. 解析响应，提取答案内容
6. 统计 Token 消耗、耗时、速度（Token/s）

## 第四部分：Function Calling（工具调用）

### 4.1 核心原理

Function Calling 让模型能够：

1. **理解用户意图**，判断是否需要调用工具
2. **从预定义工具中选择合适的工具**
3. **提取参数**，以结构化格式（JSON）返回调用指令
4. **开发者执行工具后**，将结果返回给模型继续生成

### 4.2 工具定义规范

```python
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "获取指定城市的天气信息",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {
                        "type": "string",
                        "description": "城市名称，如'北京'"
                    }
                },
                "required": ["city"]
            }
        }
    }
]
```

### 4.3 常见工具类型

| 工具类型 | 示例 | 用途 |
|:---|:---|:---|
| 文件操作 | 列出、读取、写入、删除、重命名 | 管理本地文件 |
| 网络访问 | curl 请求、API 调用 | 获取网页内容、调用第三方服务 |
| 数据库查询 | SQL 执行 | 从数据库检索信息 |
| 知识库检索 | RAG 查询 | 从向量数据库获取相关内容 |
| 时间日期 | 获取当前时间、日期计算 | 弥补模型的知识时效缺陷 |

### 4.4 工具调用完整流程

```text
用户输入 → 模型判断 → 返回工具调用指令
                ↓
        执行工具，获得结果
                ↓
        将结果加入消息历史，再次调用模型
                ↓
        模型基于工具结果生成最终回答
```

## 第五部分：RAG 与上下文管理

### 5.1 RAG（检索增强生成）

#### 工作原理

```text
用户提问 → 问题向量化 → 向量数据库检索 → 获取相关文档片段
                                              ↓
                              最终答案 ← 模型生成 ← 将检索结果作为上下文拼接到提示词中
```

#### 本地 RAG 工具：AnythingLLM

- **功能**：管理本地文档、向量化存储、提供检索 API
- **配置**：设置工作区、上传文档、开启 API 服务（默认端口 3001）
- **调用方式**：`POST http://localhost:3001/api/v1/workspace/{slug}/chat`

### 5.2 上下文压缩

当对话历史过长时，主动压缩以节省 Token 和避免超限：

- **触发条件**：对话轮数 > N 或 累计 Token > M
- **压缩策略**：对前 70% 内容进行摘要，后 30% 保留原文
- **执行方式**：调用模型生成摘要，替换原有历史

### 5.3 关键信息提取

定期从对话中提取关键信息，建立外部"记忆"：

**5W 提取规则**：
- **Who**：谁
- **What**：做了什么/说了什么
- **When**：何时（可选）
- **Where**：何处（可选）
- **Why**：为什么（可选）

**存储方式**：追加到本地日志文件，支持后续检索

## 第六部分：Skill（技能封装）

### 6.1 什么是 Skill

Skill 是将特定任务的提示词、工具调用逻辑、约束规则封装成可复用的单元。Agent 根据用户需求动态加载和调用 Skill。

### 6.2 Skill 目录结构

```text
.agents/
└── skills/
    ├── notice/
    │   └── SKILL.md
    ├── init-article/
    │   ├── SKILL.md
    │   ├── topic.md
    │   ├── voice.md
    │   ├── structure.md
    │   └── check.md
    └── code-review/
        └── SKILL.md
```

### 6.3 SKILL.md 文件格式

```markdown
---
name: 技能名称
description: 技能的详细描述，说明何时触发此技能
---

# 技能正文

## 角色定义
你是一位...

## 执行规则
1. 规则一
2. 规则二

## 输出格式
必须按照以下格式输出...

## 注意事项
- 约束条件一
- 约束条件二
```

### 6.4 Skill 加载与调用流程

1. Agent 启动时，扫描 `.agents/skills/` 目录
2. 读取每个子目录下的 `SKILL.md`，提取 YAML 中的 `name` 和 `description`
3. 将技能列表以 JSON 格式加入 System Prompt
4. 用户请求时，模型判断是否需要使用某技能
5. 如需使用，调用 `load_skill_content` 加载技能正文
6. 将技能正文加入上下文，模型按技能规则执行

### 6.5 常见 Skill 示例

| Skill 名称 | 触发场景 | 核心规则 |
|:---|:---|:---|
| 通知撰写 | 撰写/修改/润色通知 | 通知必须以"XX部通知"开头 |
| 文章生成 | 撰写读书心得、报告等 | 按 topic/voice/structure/check 四文件规范生成 |
| 代码审查 | 审查代码质量 | 检查命名、结构、安全性、性能 |

## 第七部分：链式工具调用

### 7.1 概念

链式工具调用是指 Agent 能够多步推理，将前一个工具的输出作为后一个工具的输入，自主决定调用序列，直到完成复杂任务。

### 7.2 与单次工具调用的区别

| 维度 | 单次调用 | 链式调用 |
|:---|:---|:---|
| 工具数量 | 一次调用一个工具 | 可连续调用多个工具 |
| 参数来源 | 用户输入 | 可来自上一步工具的执行结果 |
| 决策方式 | 一次性决定 | 每步都根据中间结果重新决策 |
| 应用场景 | 简单查询 | 多步骤、有依赖关系的复杂任务 |

### 7.3 链式调用核心组件

| 组件 | 职责 |
|:---|:---|
| 消息历史 | 记录每一步的对话、工具调用和结果 |
| 分析提示词 | 告知模型当前进度、历史结果、下一步可选项 |
| 决策解析器 | 解析模型返回的 JSON，判断任务是否完成 |
| 上下文管理器 | 存储中间变量，供后续工具调用使用 |
| 迭代控制器 | 防止无限循环，设置最大迭代次数 |

### 7.4 输出格式规范

```json
// 任务完成时
{"done": true, "answer": "最终回答内容"}

// 继续调用工具时
{"done": false, "tool_call": {"name": "工具名称", "arguments": {"参数名": "参数值"}}}
```

### 7.5 典型应用场景

| 场景 | 调用链示例 |
|:---|:---|
| 文件搜索 | 列出目录 → 读取文件 → 搜索关键词 → 总结结果 |
| 多文件计算 | 读文件 A → 读文件 B → 计算 → 写入结果文件 |
| 网页摘要保存 | 访问网页 → 提取正文 → 生成摘要 → 写入文件 |
| 信息汇总 | 搜索数据库 → 调用 API 补充数据 → 格式化输出 |

## 第八部分：SPEC 驱动开发

### 8.1 什么是 SPEC 范式

SPEC 是一种规范化的 AI 辅助开发方法论，通过事先定义项目规格、需求、接口和测试计划，提高开发质量和效率。

### 8.2 四大核心文档

| 文件 | 内容 | 作用 |
|:---|:---|:---|
| spec.md | 项目目标、范围、用户故事 | 明确"做什么" |
| requirement.md | 详细功能和非功能需求 | 细化"具体要求" |
| api.md | 函数/模块的输入、输出、伪代码 | 定义"怎么做" |
| test.md | 测试用例和验收标准 | 验证"做对了没" |

### 8.3 SPEC 开发流程

1. **需求讨论**：与 AI 对话，明确需求和边界
2. **生成 spec.md**：定义项目目标和范围
3. **生成 requirement.md**：细化功能需求
4. **生成 api.md**：设计接口和伪代码
5. **生成 test.md**：规划测试用例
6. **逐文件审阅确认**
7. **开发**：AI 根据 spec/requirement/api 进行编码
8. **测试**：根据 test.md 进行测试和修复

### 8.4 规范文档生成 Skill

将 SPEC 流程封装成 `init-spec` Skill，输入一句话需求，自动生成四份规范文档。

**复合文档生成 Skill**（如 `init-article`）：

| 文件 | 规范内容 |
|:---|:---|
| topic.md | 主题范围、禁止偏离的内容 |
| voice.md | 语气、风格、人称、专业程度 |
| structure.md | 章节结构、每个章节的要点 |
| check.md | 字数、格式、关键要素检查清单 |
