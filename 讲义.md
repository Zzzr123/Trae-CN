## 第一章：AI Agent 核心基础概念

本章是后续所有实操开发的**理论基石**，讲解 AI 智能体开发必备的底层核心概念，这些概念将贯穿整个 Agent 开发体系。

### 1.1 核心交互概念

#### 1.1.1 System Prompt（系统提示词）
- **定义**：结构化的固定指令，用于给大模型定义角色、行为规则、能力边界、输出格式与约束逻辑
- **作用**：决定模型的基础行为模式，全局生效
- **类比**：就像给 AI 设定一个"身份说明书"，告诉它该做什么、不该做什么

#### 1.1.2 User Prompt（用户提示词）
- **定义**：用户实时输入的自然语言指令
- **作用**：触发单次交互的核心输入，用于发起具体任务需求
- **特点**：时效性强，仅对当前对话轮次有效

#### 1.1.3 Token（词元）
- **定义**：大模型运算的最小单位，中英文文字、标点、符号、空格均会占用 Token
- **作用**：统计模型消耗、限制上下文长度、控制调用成本
- **提示**：理解 Token 概念对后续控制成本和优化性能至关重要

#### 1.1.4 Context（上下文）
- **定义**：多轮对话的历史内容集合
- **作用**：实现连贯对话、理解完整需求的核心机制
- **权衡**：上下文越长，对话连贯性越强，但资源消耗和 Token 占用越高

### 1.2 核心技术架构概念

#### 1.2.1 RAG（检索增强生成）
- **定位**：解决大模型知识滞后、无法读取私有数据的核心技术
- **原理**：通过检索本地私有文档、外部实时数据，辅助模型生成精准、真实、实时的回答
- **价值**：企业级 AI 应用的基础技术，让 AI 能够"知新"和"知私"

#### 1.2.2 Function Call（工具调用）
- **定位**：大模型突破自身能力边界的核心机制
- **原理**：大模型作为逻辑决策核心，通过调用外部自定义代码、工具、网络接口、文件操作完成实操任务
- **意义**：是 AI 从"对话"走向"做事"的关键一步

#### 1.2.3 Skill（技能）
- **定位**：可插拔、标准化、可复用的 AI 能力模块
- **特点**：开发者可将高频功能封装为独立 Skill，Agent 可动态扫描、按需加载、自动调用
- **优势**：实现能力模块化扩展，降低代码耦合度

#### 1.2.4 MCP（智能体通信协议）
- **定位**：智能体模块化通信与调用协议
- **作用**：规范 Agent 内部各模块、技能、工具之间的调用逻辑、数据传输与协同规则
- **价值**：保证多模块稳定协作，支持复杂系统开发

#### 1.2.5 Agent（智能体）
- **定义**：具备自主感知、记忆存储、逻辑规划、工具调用、迭代优化能力的自主 AI 程序
- **核心区别**：区别于普通单轮对话 AI，可无需人工干预，自主完成复杂多步骤任务
- **目标状态**：像一个聪明的"数字员工"，能够独立完成指定任务

## 第二章：提示工程核心技术

提示工程是所有 AI 应用的**基础技能**，核心是通过规范化指令，约束大模型的输出风格、结构、逻辑与精度，最大化发挥模型能力。

### 2.1 基础核心提示技巧

#### 2.1.1 双角色精准定义
- **方法**：同时明确 AI 执行角色与用户身份
- **效果**：精准限定输出的专业度、文风、视角，避免输出内容泛化

#### 2.1.2 结构化输出约束
- **方法**：提前定义输出格式（段落、列表、表格、框架）
- **效果**：解决模型输出杂乱、格式不统一的问题

#### 2.1.3 双 AI 辩论优化
- **方法**：通过对抗式提问、多角度校验
- **效果**：让模型自我修正逻辑漏洞、优化内容结构、补齐信息短板

#### 2.1.4 上下文手动构建
- **方法**：人工拼接参考资料、对话历史、需求背景
- **效果**：为模型提供完整上下文，保证精准理解复杂需求

#### 2.1.5 轻量化上下文压缩
- **方法**：通过摘要、精简、去重等方式压缩历史对话
- **效果**：在保留核心信息的前提下，减少 Token 消耗，避免上下文溢出

### 2.2 进阶精细化提示技巧

#### 2.2.1 反向约束提示
- **方法**：在正向需求基础上，明确禁止内容、规避错误输出、限定内容边界
- **特点**：支持段落、列表、表格三种输出格式自由切换

#### 2.2.2 AI 主动交互补全
- **场景**：解决用户需求模糊、信息不全的问题
- **模式**：
  - 批量提问：一次性补齐所有缺失信息
  - 单次单问：提供选项，降低沟通成本

### 2.3 AI 风格复刻生成技术
- **能力**：支持以文字、图片、网页、文档等任意素材为参考
- **效果**：让 AI 精准复刻原有风格、结构、逻辑框架，完全原创生成新内容
- **应用场景**：文稿创作、PPT 设计、项目文案、竞赛材料制作

### 2.4 提示工程落地实验（建议实操）

通过多组对照实验，直观验证提示词的作用效果：

| 实验名称 | 实验目标 | 验证内容 |
|---------|---------|---------|
| 角色定义对照实验 | 验证角色设定的影响 | 不同角色设定对输出专业度、文风、内容侧重点的影响 |
| 框架约束对照实验 | 验证结构约束的价值 | 有无输出框架约束下，内容规范性、逻辑性、完整性的差异 |
| 长文本生成实验 | 综合应用提示技巧 | 运用全套提示技巧，稳定完成万字级长文本连贯创作 |
| 场景化实战实验 | 真实场景落地 | AI 辅助完成方案设计、文稿撰写、视觉风格定制全流程 |

## 第三章：大模型参数解析与本地 LLM 部署

本地大模型部署是 AI Agent **本地化、私有化、低成本**开发的核心，掌握模型参数调优，可适配不同电脑配置，实现最优运行效果。

### 3.1 大模型核心运行参数详解

#### 3.1.1 模型规模（xxB）
- **含义**：代表模型参数总量
- **权衡**：数值越大，推理能力越强，但显存占用越高

#### 3.1.2 MoE 架构（A3B）
- **含义**：混合专家架构
- **优势**：动态激活部分模型参数，平衡性能与速度，适配中端设备

#### 3.1.3 量化精度（Q4/Q6）
- **Q4 量化**：压缩比例高，体积小，适合低配设备
- **Q6 量化**：精度更高，输出效果更精准，适合追求质量的场景

#### 3.1.4 Context Length（上下文长度）
- **含义**：模型可一次性处理的最大文本篇幅
- **影响**：长度越高，长文本处理能力越强

#### 3.1.5 KV Cache（键值缓存）
- **作用**：缓存对话历史的计算结果，避免重复计算
- **效果**：大幅提升对话生成速度，降低算力消耗

#### 3.1.6 KV 缓存 GPU 卸载
- **作用**：将 KV 缓存数据转移至 GPU 内存运行
- **效果**：释放 CPU 压力，显著提升推理速度

### 3.2 多设备本地模型部署方案

| 设备类型 | 推荐方案 | 特点 |
|---------|---------|------|
| Nvidia 独显设备 | LMStudio + Qwen 系列模型 | 高速离线运行，需保证显存充足 |
| AMD 显卡/轻薄本/无独显 | CPU 模式运行 LMStudio 或在线平台 | 速度较慢，但可完成开发与实验 |

### 3.3 首个本地 LLM 调用项目开发（实操步骤）

**核心开发流程：**

1. **环境校验**
   - 配置 Python 3.12+ 环境
   - 确保 LMStudio 正常加载本地模型
   - 项目路径无中文、无空格

2. **项目初始化**
   - 新建空白项目目录
   - 配置虚拟环境 venv
   - 生成 .gitignore 文件

3. **配置文件设计**
   - 创建 env.example 模板文件
   - 预留 Base_URL、Model、Token 等配置项
   - 支持自定义私有 .env 配置

4. **核心代码开发**
   - 读取本地配置
   - HTTP 请求调用本地 LLM
   - 统计 Token 消耗、响应时长、生成速度

5. **参数调优迭代**
   - 反复调试模型加载参数
   - 测试出最稳定、最高效的运行配置

## 第四章：多轮对话系统与基础工具调用开发

本章实现从**单次模型调用**到**可持续多轮交互 + 工具能力拓展**的升级，让 AI 具备持续对话和实操能力。

### 4.1 流式多轮对话系统开发

**核心功能：**

- **流式输出**：文字逐字实时刷新，模拟真人对话效果
- **历史上下文记忆**：自动拼接多轮对话历史，保证逻辑连贯
- **循环交互机制**：程序持续运行，支持多次输入（Ctrl+C 终止）
- **项目标准化管理**：编写 README.md，配置 Git 版本控制

### 4.2 五大基础文件工具 Function Call

为 Agent 赋予本地文件自主操作能力：

| 工具名称 | 功能描述 | 核心能力 |
|---------|---------|---------|
| 目录查询工具 | 读取指定目录下所有文件 | 返回文件名称、大小、格式、修改时间 |
| 文件重命名工具 | 修改指定路径下的文件名称 | 支持自定义命名规则 |
| 文件删除工具 | 删除目标目录下的指定文件 | 安全删除，规避误删风险 |
| 文件创建写入工具 | 新建文件并写入内容 | 支持文本、代码、数据等内容 |
| 文件读取工具 | 读取本地任意文件内容 | 为分析、总结、编辑提供数据支撑 |

### 4.3 网络工具开发与模型缺陷优化

#### 4.3.1 网络访问工具
- **实现方式**：基于 curl 开发外网请求功能
- **能力扩展**：让 Agent 具备主动访问外网 API、获取实时网络数据的能力

#### 4.3.2 模型固有缺陷修复
- **问题**：大模型无法自主获取实时时间、实时热点等动态信息
- **解决方案**：开发日期获取函数，或在 System Prompt 中自动附加当前时间

### 4.4 长文本实战测试
- **任务**：执行长文本连载生成任务，自动保存至本地文件
- **验证目标**：工具调用稳定性、长文本生成连贯性、程序运行可靠性

## 第五章：RAG 知识库与智能记忆系统开发

本章实现 Agent **私有文档认知 + 智能上下文优化 + 长期记忆**能力，解决模型知识滞后、上下文溢出、无记忆能力的核心痛点。

### 5.1 AnythingLLM 本地向量知识库部署

**核心流程：**

1. **环境绑定**：安装 AnythingLLM，关联 LMStudio 本地大模型
2. **工作区创建**：新建专属工作区，上传本地私有文档、资料、笔记
3. **能力落地**：实现文档向量检索、智能问答、片段匹配、轻量化上下文压缩

### 5.2 智能上下文自动压缩系统

**触发条件**：
- 对话轮次超过 5 轮 **或** 上下文 Token 长度超过 3000

**压缩规则**：
- 保留后 30% 最新原始对话内容
- 对前 70% 历史内容进行 AI 摘要压缩
- 兼顾对话连贯性与轻量化

### 5.3 基于 5W 规则的长期记忆系统

**信息提取规则（每 5 轮对话自动触发）：**
- **Who**：人物
- **What**：事件
- **When**：时间（可选）
- **Where**：场景（可选）
- **Why**：原因（可选）

**存储与检索**：
- 自动创建本地日志目录与文件
- 持续增量沉淀对话关键信息
- 支持自定义指令触发历史查询

## 第七章：Skill 可插拔技能系统开发

Skill 技能系统是**高级 Agent 的核心特性**，实现 AI 能力模块化、可复用、可动态扩展。

### 7.1 Skill 技能核心原理

- **定义**：标准化、轻量化、可插拔的 AI 功能模块
- **开发模式**：将高频通用功能封装为独立技能，存放于项目指定目录
- **使用方式**：Agent 自动扫描技能库、识别技能功能、根据用户需求自主加载调用

### 7.2 技能自动加载系统开发

**两大核心函数：**

1. **list_available_skills（技能扫描函数）**
   - 自动遍历项目 .agents/skills 目录
   - 读取 SKILL.md 配置文件
   - 提取技能名称、功能描述
   - 以 JSON 格式注入系统提示词

2. **load_skill_content（技能加载函数）**
   - 当大模型判定需要使用某一技能时触发
   - 读取对应 SKILL.md 的完整规则内容
   - 约束 AI 按照技能规则执行任务

### 7.3 自定义 Skill 实战落地（通知生成技能案例）

**技能规则定义：**
- 通知文案禁止以"通知"二字开头
- 必须携带部门前缀
- 无指定部门默认使用"XX部"

**场景测试：**
- 场景 1：无部门指定 → 生成"XX部放假通知"
- 场景 2：指定销售部 → 生成"销售部通知"

## 第八章：Agent 链式工具调用（自主决策系统）

链式工具调用是 AI Agent 实现**完全自主智能化**的核心标志，让 AI 无需人工干预，自主规划任务步骤、连续调用多工具完成复杂任务。

### 8.1 链式调用核心原理

**传统调用 vs 链式调用：**

| 维度 | 传统工具调用 | 链式工具调用 |
|-----|------------|-------------|
| 调用方式 | 单次单工具调用 | 多工具连续调用 |
| 人工干预 | 需要人工指令 | 无需人工干预 |
| 数据流转 | 无自动传递 | 前一工具输出作为后一工具输入 |
| 适用场景 | 简单任务 | 复杂多步骤任务 |

### 8.2 链式调用核心架构开发

**三大核心组件：**

1. **ChainedCallContext（上下文管理类）**
   - 记录每一步工具调用日志
   - 存储中间结果
   - 管理任务状态
   - 设置最大迭代次数（默认 10 次），防止无限循环

2. **execute_chained_tool_call（执行主函数）**
   - 消息初始化
   - 循环决策
   - 工具执行
   - 结果回写
   - 任务终止判断

3. **build_analysis_prompt（决策提示词函数）**
   - 整合用户原始需求
   - 整合历史执行步骤
   - 整合决策规则
   - 引导大模型自主判断

### 8.3 标准化决策输出格式

**任务完成格式：**
```json
{"done": true, "answer": "最终回答内容"}
```

**继续执行格式：**
```json
{"done": false, "tool_call": {"name": "工具名", "arguments": {"参数名": "参数值"}}}
```

### 8.4 三大实战场景验证

| 场景 | 任务描述 | 验证目标 |
|-----|---------|---------|
| 文件智能分析 | 遍历目录、筛选含指定关键词的代码文件、读取内容并总结 | 多工具协作能力 |
| 多文件数据联动 | 读取两个数值文件、完成数据运算、创建新文件写入结果 | 数据流转能力 |
| 网页信息萃取 | 访问外网新闻页面、提取核心正文、保存为本地文档 | 网络访问与数据处理能力 |

### 8.5 容错与兼容机制

- **JSON 代码块解析**：支持 Markdown 代码块格式
- **多格式响应兼容**：支持 tool_calls 格式和 JSON content 格式
- **空值异常捕获**：处理 LLM 响应为 None 的情况
- **迭代次数限制**：防止无限循环

## 第九章：SPEC 工程化开发范式

SPEC 是标准化 AI 项目开发流程，通过四份规范文档约束开发全流程，解决需求模糊、代码混乱、无测试标准、迭代无序的问题。

### 9.1 SPEC 范式核心价值

- 将传统随性开发升级为标准化、流程化、可追溯、可测试的工程化开发
- 适配所有 AI 智能体项目、工具类项目、创意开发项目
- 大幅提升项目规范性与可维护性

### 9.2 四大核心规范文档
